# 需求探索文档 · astrology（网页占星应用）

> 阶段 1（需求探索）产出物。把"一句话需求"澄清成有边界、无歧义的共识。后续所有阶段复用本文档的统一语言与范围界定。

---

## 1. 背景与目标

### 要解决的问题
对占星感兴趣的普通个人用户，缺少一个"一站式、无门槛"的中文网页占星工具：现有工具要么需要注册、要么信息割裂（星盘、运势、知识分散在不同产品）、要么专业术语晦涩、缺少解读。本产品要让用户**输入出生信息后即可获得完整可视化本命星盘，并结合每日/每周运势与内置百科加深对自身的了解**。

### 核心价值主张
- 输入出生信息 → 一键生成可视化本命盘。
- 星盘 + 运势 + 知识百科三合一，降低占星入门门槛。
- 无需注册、免费、开箱即用，暗色神秘视觉营造沉浸感。

### 成功长什么样（可衡量的目标形态）
- 新用户在**不注册**的前提下，从进入首页到看到自己的本命盘 ≤ 1 分钟、≤ 3 个必填字段（出生日期、出生时间、出生地点）。
- 星盘正确渲染十大行星、十二宫、十二星座、上升点，且对同一出生信息结果稳定可复现。
- 用户可查看"当前日期"的每日运势与本周运势，并能在百科中查到任一行星/星座的解释。
- 页面在主流桌面与移动浏览器上均可正常访问与阅读（响应式）。

> 说明：具体运营指标（DAU/留存等）不属于本阶段范围，见"未决问题 UQ-8"。

## 2. 用户与角色

| 角色 | 描述 | 关心什么 |
|------|------|----------|
| 占星爱好者（主要用户，访客） | 对占星感兴趣的普通个人，未必懂专业术语。无需登录即可使用。 | 快速看到自己的星盘、每天有没有新鲜运势、看不懂的名词能不能查到解释 |
| 好奇的初次访客 | 被朋友分享或搜索进来，可能只想试一次 | 不用注册、不用填太多、能立刻出结果 |
| 内容/知识维护者（内部角色） | 维护百科词条与运势文案的人（可能是运营，也可能是预置静态内容） | 词条准确、运势文案可更新、术语一致 |
| 系统本身（自动参与者） | 到达每日/每周节点时自动切换当日/当周运势内容 | 运势按日期正确滚动、不重复陈旧 |

> 本阶段暂不引入"注册用户/会员"角色，见"未决问题 UQ-1"。

## 3. 核心业务事件（事件风暴）

按业务时间线梳理，格式：**领域事件（过去式）← 触发命令 ← 角色｜涉及外部系统/依赖｜产生的读模型**

### 主流程 A：生成本命盘
| # | 领域事件 | 触发命令 | 角色 | 外部系统/依赖 | 读模型/界面 |
|---|----------|----------|------|----------------|-------------|
| A1 | 出生信息已提交 | 输入并提交出生日期/时间/地点 | 占星爱好者 | 地点→经纬度（地理编码）；时区判定 | 出生信息表单、校验提示 |
| A2 | 出生地点已定位 | （由 A1 自动触发）解析地点为经纬度与时区 | 系统 | 地理编码服务、时区数据 | 地点候选/确认 |
| A3 | 行星位置已计算 | （由 A2 自动触发）依据时刻与坐标计算天体位置 | 系统 | 天文历书/星历计算能力 | —（中间结果） |
| A4 | 本命盘已生成 | （由 A3 自动触发）汇总为可视化星盘 | 系统 | — | 星盘图（宫位/行星/星座/相位）、行星落座列表 |
| A5 | 星盘要素已解读 | 查看某行星/宫位/上升的含义 | 占星爱好者 | 百科内容库 | 解读文案、要素详情卡 |

### 主流程 B：查看运势
| # | 领域事件 | 触发命令 | 角色 | 外部系统/依赖 | 读模型/界面 |
|---|----------|----------|------|----------------|-------------|
| B1 | 每日运势已呈现 | 打开每日运势/选择星座 | 占星爱好者 | 运势内容来源（见 UQ-4） | 今日运势卡（爱情/事业/健康等维度） |
| B2 | 每周运势已呈现 | 切换到每周运势视图 | 占星爱好者 | 运势内容来源 | 本周运势卡 |
| B3 | 运势已按日期滚动 | （系统在日期/周切换时自动）更新当日/当周内容 | 系统 | 当前日期/时钟 | 最新日期标签、当期运势 |

### 主流程 C：浏览百科
| # | 领域事件 | 触发命令 | 角色 | 外部系统/依赖 | 读模型/界面 |
|---|----------|----------|------|----------------|-------------|
| C1 | 百科词条已展示 | 浏览/搜索某星座或行星 | 占星爱好者 | 百科内容库 | 词条详情（符号、象征、关键词、解释） |
| C2 | 词条已检索 | 输入关键词搜索 | 占星爱好者 | 百科内容库 | 搜索结果列表 |
| C3 | 百科内容已更新 | 新增/修改词条 | 内容维护者 | 百科内容库 | 词条管理界面（视 UQ-6 而定） |

### 说明
- 上表中命令均为"用户意图/动作"，事件为"系统里已发生的事实"，两者一一对应，供后续 BDD 场景与式样追溯。
- A2/A3/A4 为链式自动事件：一次"提交出生信息"命令，触发一连串系统事件，最终以"本命盘已生成"对用户可见。

## 4. 外部系统 / 依赖（业务视角）

> 均为"能力需求"，非具体技术选型（选型属阶段 5）。此处仅明确"需要一种什么样的外部能力"。

| 依赖能力 | 用途 | 业务约束 | 关联未决问题 |
|----------|------|----------|--------------|
| 地理编码（地点 → 经纬度） | 把用户输入的出生城市转为坐标，用于星盘计算 | 优先免费/无需注册；覆盖中文城市名 | UQ-3 |
| 时区判定 | 由坐标+日期得到出生时刻的时区（含历史夏令时） | 影响星盘准确性，需可靠 | UQ-3 |
| 天体位置计算（星历） | 计算十大行星、上升点等的黄道位置 | 需可复现、精度足够（分钟级出生时间敏感） | UQ-2 |
| 运势内容来源 | 每日/每周运势文案 | 需明确是第三方运势 API 还是自有/预置文案 | UQ-4 |
| 百科内容库 | 行星/星座词条 | 中文、准确、术语与统一语言一致 | UQ-5 |

## 5. 读模型 / 关键界面（用户为做决策需要看到什么）

- **出生信息表单**：日期、时间、地点三项输入 + 校验提示（时间未知时的处理见 UQ-7）。
- **本命盘视图**：圆形星盘图（黄道十二宫、十二星座刻度、十大行星符号及落座、上升/中天点、主要相位连线）+ 行星落座清单（行星/星座/宫位）。
- **星盘要素解读卡**：点击某行星/星座/宫位弹出其含义解释（与百科打通）。
- **每日运势视图**：按星座呈现今日运势，含多维度（如爱情/事业/健康/幸运色等，最终维度见 UQ-4）。
- **每周运势视图**：本周整体运势。
- **百科首页 + 词条详情**：星座列表、行星列表、搜索框、词条详情页。
- **全局导航**：在"星盘 / 运势 / 百科"三大板块间切换。

## 6. 范围（Scope）

### ✅ In scope（本产品要做）
1. 出生本命盘生成：输入出生日期、时间、地点，生成可视化星盘（十大行星、十二宫、十二星座、上升点、主要相位）。
2. 星盘要素基础解读（与百科联动的行星落座/星座/宫位含义）。
3. 每日运势与每周运势的展示（按星座）。
4. 行星/星座百科：词条浏览、搜索、详情。
5. 神秘暗色系响应式网页界面（深紫/深蓝背景 + 金色/星光装饰），支持桌面与移动浏览器。
6. 无需注册即可使用全部核心功能（访客模式）。

### ❌ Out of scope（本版本明确不做）
1. 用户账号体系、登录注册、云端保存历史星盘（除非 UQ-1 确认要做）。
2. 合盘（两人星盘对比）、推运/流年、择时、占卜（塔罗等）等进阶占星功能。
3. 付费/订阅/会员、广告变现。
4. 社交功能（分享到社区、评论、关注）—— 单纯的"分享链接"是否保留见 UQ-8。
5. AI 生成个性化长篇解读（本版本运势/解读以预置或规则化文案为主，见 UQ-4）。
6. 原生 App（iOS/Android）、桌面客户端。
7. 多语言国际化（首版仅中文，见 UQ-9）。
8. 通知推送（邮件/短信/App push）——"每日/每周运势推送"在本版本理解为"用户打开即见当期内容"，而非主动推送，见 UQ-4。

## 7. 统一语言 · 术语表

| 术语 | 含义（本项目统一定义） |
|------|------------------------|
| 本命盘（出生星盘 / Natal Chart） | 依据个人出生的精确日期、时间、地点绘制的天体位置图，反映出生瞬间天空的行星分布 |
| 星盘 | 泛指以圆形黄道为底、标注行星与宫位的占星图，本项目默认指"本命盘" |
| 出生信息 | 生成本命盘所需的三要素：出生日期、出生时间、出生地点 |
| 行星（天体） | 占星使用的十大天体：太阳、月亮、水星、金星、火星、木星、土星、天王星、海王星、冥王星（含日月，占星语境统称"行星"） |
| 行星位置 / 落座 | 某行星在星盘中所处的黄道星座与宫位 |
| 黄道十二星座 | 白羊、金牛、双子、巨蟹、狮子、处女、天秤、天蝎、射手、摩羯、水瓶、双鱼 |
| 太阳星座 | 出生时太阳所在的星座，即大众常说的"我是 X 座" |
| 上升星座（Ascendant / ASC） | 出生时刻东方地平线正在升起的星座，依赖精确出生时间与地点 |
| 中天（MC） | 星盘中天顶所在点，与事业/社会形象相关 |
| 宫位（十二宫） | 星盘划分的十二个生活领域区块（如第一宫=自我，第七宫=关系等） |
| 相位（Aspect） | 行星之间的角度关系（如合相 0°、对分 180°、三分 120° 等），主要相位在星盘中以连线表示 |
| 星历 / 天体历书（Ephemeris） | 提供指定时刻天体黄道位置的数据/算法来源，是星盘计算的基础 |
| 地理编码 | 将"出生地点"（城市名）转换为经纬度坐标的过程 |
| 每日运势 | 面向某星座、针对"当日"的运势描述，可含多个维度 |
| 每周运势 | 面向某星座、针对"本周"的运势描述 |
| 运势维度 | 运势内容拆分的方面，如爱情、事业、财运、健康、幸运色/数字（最终维度待定，见 UQ-4） |
| 百科 / 词条 | 关于某个行星或星座的知识条目，含符号、象征、关键词与解释 |
| 访客模式 | 无需注册登录即可使用全部核心功能的状态 |
| 当期运势 | 依据当前系统日期自动对应的"今日/本周"运势 |

## 8. 关键风险与未决问题

> 以下为信息不足或存在歧义之处。**不自行编造业务规则**，交由使用者/后续门禁确认。每项给出"默认假设"，使用者不答时按默认推进（标注"假设"）。

| 编号 | 未决问题 | 默认假设（未答即采用） | 影响 |
|------|----------|------------------------|------|
| UQ-1 | 是否需要账号体系以保存历史星盘（如为家人朋友保存多张盘）？ | ✅ **已确认**：**不做**账号，纯访客；星盘信息不持久保存 | 影响架构、Out-of-scope 第 1 条 |
| UQ-2 | 星历计算精度与来源：用高精度星历（如 Swiss Ephemeris 级）还是够用的简化算法？ | ✅ **已确认**：优先**离线/本地 JS 库**方案，精度满足十大行星+上升即可 | 影响星盘准确性与选型（阶段5） |
| UQ-3 | 地理编码 + 时区来源？是否允许离线内置城市库以避免依赖付费/注册 API？ | ✅ **已确认**：优先**内置城市库或离线本地方案** | 影响准确性与"免费无注册"目标 |
| UQ-4 | 运势内容来源：第三方运势 API、自有编写文案、还是规则/模板生成？"推送"是主动通知还是打开即见？ | ✅ **已确认**：接入**第三方运势 API**；打开即见（非主动推送） | 影响 Out-of-scope 第 5/8 条、外部依赖；选型阶段需调研可用免费 API |
| UQ-5 | 百科内容由谁提供、是否需要审校？覆盖范围是仅十二星座+十大行星，还是含宫位/相位词条？ | 假设：首版覆盖**十二星座 + 十大行星**，内容为预置静态文案；宫位/相位解释作为解读卡内联，不单列词条 | 影响内容工作量与界面 |
| UQ-6 | 百科/运势内容是否需要后台管理界面供维护者更新，还是直接改静态数据文件即可？ | 假设：首版**无后台**，内容以静态数据维护 | 影响 C3 事件、角色"内容维护者" |
| UQ-7 | 用户不知道精确出生时间怎么办（会影响上升星座/宫位）？ | 假设：允许"时间未知"，此时**给出太阳星座与行星落座，但标注上升/宫位不可用或按正午估算并提示** | 影响表单与星盘可用性 |
| UQ-8 | 是否需要"生成分享链接/图片"让用户把星盘发给别人？ | 假设：首版**不做**分享，仅本地查看 | 影响 Out-of-scope 第 4 条 |
| UQ-9 | 语言与受众：确认首版仅中文、面向国内用户？ | 假设：**仅中文**，单语言 | 影响国际化范围 |
| UQ-10 | 免责声明：占星内容属娱乐/参考，是否需在页面明示免责，避免被当作医疗/投资建议？ | 假设：**需要**在运势/解读处加"仅供娱乐参考"免责声明 | 合规与信任 |

### 关键风险
- **准确性风险**：星盘正确性高度依赖出生时间精度、时区/夏令时处理与星历质量（UQ-2/UQ-3/UQ-7）；错误结果会直接损害信任。
- **依赖可得性风险**："免费/无需注册"目标可能与高质量地理编码/运势 API 冲突，需在选型阶段权衡（UQ-3/UQ-4）。
- **内容质量风险**：运势与百科为文字内容，准确性与一致性依赖内容来源（UQ-4/UQ-5）。
- **合规/期望管理风险**：占星内容需明确免责定位（UQ-10）。

## 附：本阶段结论
- 领域可清晰划分为**星盘生成 / 运势 / 百科**三个子域，彼此低耦合，适合分阶段实装。
- 核心不确定性集中在**外部依赖选型**（星历、地理编码、运势/内容来源），已全部列入未决问题，留待阶段 5 选型或使用者确认，本阶段**不预设技术方案、不编造业务规则**。
- 建议在门禁评审时优先确认 UQ-1、UQ-4、UQ-2/UQ-3，因其直接决定范围与架构走向。
